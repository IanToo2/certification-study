# 2장. Compute

컴퓨트 영역은 시험에서 가장 넓게 출제된다. `Amazon EC2`, 오토스케일링, 서버리스, 컨테이너, 관리형 플랫폼이 어떤 문제에 맞는지 구분해야 한다. 핵심은 운영 부담, 확장 단위, 실행 시간, 상태 저장 여부다.

## Amazon EC2

### 1. 서비스 개념

`Amazon EC2`는 AWS의 대표적인 가상 서버 서비스다. 운영체제 수준 제어권이 필요하거나, 특정 런타임과 에이전트 설치가 필요한 워크로드에 사용한다.

### 2. 아키텍처 구조

```text
Client
  |
  v
ALB
  |
  v
Auto Scaling Group
  |
  +--> EC2 in AZ-a
  +--> EC2 in AZ-b
         |
         v
        EBS
```

### 3. 언제 사용하는가

- OS 접근, 커널 설정, 사용자 정의 에이전트 설치가 필요할 때
- 장기 실행 프로세스나 상태 기반 애플리케이션을 운영할 때
- 리프트 앤 시프트 마이그레이션이 필요할 때

### 4. 언제 사용하면 안되는가

- 짧고 이벤트 기반인 함수형 워크로드일 때
- 서버 관리 없이 코드를 실행하는 것이 핵심 요구사항일 때

### 5. 비슷한 서비스와 차이

- `AWS Lambda`는 서버리스, `Amazon EC2`는 서버 관리 필요
- `AWS Elastic Beanstalk`는 `EC2` 기반을 자동화하지만 세부 제어는 `EC2` 직접 운영보다 적다
- `Amazon ECS`/`AWS Fargate`는 컨테이너 단위 배포에 적합하다

### 6. 시험에서 자주 나오는 문제 유형

- 레거시 애플리케이션을 빠르게 AWS로 이전하는 문제
- 사용자 데이터, AMI, 보안 그룹, EBS 스냅샷을 묻는 문제
- 고가용성을 위해 Multi-AZ에 `EC2`를 배치하는 문제

### 7. 시험 함정 포인트

- `EC2` 자체는 관리형 고가용성이 아니다. `Auto Scaling Group`, `ALB`, Multi-AZ와 함께 봐야 한다.
- 인스턴스 종료 시 루트 EBS 보존 여부를 묻는 세부 문제가 자주 나온다.

### 8. 실무 아키텍처 예시

전자상거래 웹 서버를 `ALB` 뒤의 `Auto Scaling Group`으로 운영하고, 세션 상태는 `ElastiCache`, 데이터는 `Amazon RDS`에 저장한다. 인스턴스는 무상태에 가깝게 유지해야 확장과 교체가 쉽다.

### 시험 핵심 포인트

- 운영체제 제어, 에이전트 설치, 레거시 앱 이전 요구가 있으면 `Amazon EC2`를 먼저 본다.
- 고가용성은 `EC2` 단독이 아니라 `ALB + Auto Scaling + Multi-AZ` 조합으로 판단한다.
- 서버 관리 최소화 요구가 강하면 `AWS Lambda`나 `AWS Fargate`가 더 적합할 수 있다.

## EC2 Instance Types

### 1. 서비스 개념

`EC2 Instance Types`는 CPU, 메모리, 스토리지, 네트워크 성능 조합을 미리 정의한 인스턴스 패밀리다. 시험에서는 워크로드 특성에 맞는 패밀리 선택이 중요하다.

### 2. 아키텍처 구조

```text
Workload Requirement
   |
   +--> General Purpose (balanced)
   +--> Compute Optimized (CPU-heavy)
   +--> Memory Optimized (RAM-heavy)
   +--> Storage Optimized (high IOPS/local disks)
   +--> Accelerated Computing (GPU/FPGA)
```

### 3. 언제 사용하는가

- CPU 집약형 배치 작업이면 `C` 패밀리
- 메모리 캐시나 인메모리 DB면 `R` 또는 `X` 패밀리
- 균형형 웹 애플리케이션이면 `M` 또는 `T` 패밀리

### 4. 언제 사용하면 안되는가

- 인스턴스 유형을 무작정 크게 키워 아키텍처 문제를 해결하려는 경우
- 트래픽 변동이 큰데 고정 대형 인스턴스로만 대응하려는 경우

### 5. 비슷한 서비스와 차이

- `T` 계열은 버스트 크레딧 기반이라 상시 고CPU에는 부적합하다.
- `M`은 범용, `C`는 CPU, `R`은 메모리 중심으로 선택 기준이 분명하다.

### 6. 시험에서 자주 나오는 문제 유형

- CPU 100% 지속인 워크로드에 `T3`를 고르게 유도하는 오답
- 메모리 부족 애플리케이션에서 `C` 대신 `R`을 선택해야 하는 문제

### 7. 시험 함정 포인트

- 버스트 성능과 지속 성능을 구분해야 한다.
- 비용 최적화 문제에서 과대 스펙 인스턴스는 흔한 오답이다.

### 8. 실무 아키텍처 예시

백오피스 API는 `M` 패밀리, 실시간 이미지 변환은 `C` 패밀리, Redis 캐시는 `R` 패밀리로 분리하면 비용과 성능의 균형이 좋아진다.

### 시험 핵심 포인트

- CPU 중심이면 `C`, 메모리 중심이면 `R`, 균형형이면 `M` 계열을 우선 떠올린다.
- `T` 계열은 버스트 성능이므로 상시 고부하 문제에 자주 오답으로 나온다.
- 비용 최적화 문제에서는 과대 스펙 인스턴스를 피해야 한다.

## EC2 Placement Groups

### 1. 서비스 개념

`EC2 Placement Groups`는 인스턴스의 물리적 배치 전략을 제어하는 기능이다. `Cluster`, `Spread`, `Partition` 세 가지가 시험 핵심이다.

### 2. 아키텍처 구조

```text
Placement Group
  |
  +--> Cluster: same rack proximity
  +--> Spread: distinct hardware
  +--> Partition: isolated partitions
```

### 3. 언제 사용하는가

- 초저지연 고성능 통신이 필요하면 `Cluster`
- 소수의 중요 인스턴스를 물리적으로 분리하려면 `Spread`
- 대규모 분산 시스템에서 랙 단위 장애 격리가 필요하면 `Partition`

### 4. 언제 사용하면 안되는가

- 단순 웹 서버처럼 물리 배치 제어가 큰 의미가 없을 때
- 리전 간 분산을 Placement Group으로 해결하려는 경우

### 5. 비슷한 서비스와 차이

- `Cluster`는 성능 우선, 장애 도메인 분리에는 불리하다.
- `Spread`는 내결함성 우선이지만 인스턴스 수 제한이 있다.
- `Partition`은 Hadoop, Kafka 같은 대규모 분산 시스템에 적합하다.

### 6. 시험에서 자주 나오는 문제 유형

- 고성능 컴퓨팅에 가장 낮은 네트워크 지연을 요구하는 문제
- 하드웨어 장애 전파를 최소화해야 하는 문제

### 7. 시험 함정 포인트

- Placement Group은 AZ 내부 물리 배치 관점이지, 멀티 리전 DR 전략이 아니다.
- `Cluster`가 항상 고가용성에 좋은 것은 아니다.

### 8. 실무 아키텍처 예시

거래 분석용 HPC 노드는 `Cluster Placement Group`으로 묶고, 컨트롤 노드는 별도 그룹으로 분리해 성능과 안정성을 균형 있게 가져간다.

### 시험 핵심 포인트

- 초저지연 고성능 통신이면 `Cluster Placement Group`이 정답 후보다.
- 장애 도메인 분리가 중요하면 `Spread` 또는 `Partition`을 검토한다.
- 성능 최적화와 고가용성 목적이 서로 다를 수 있다는 점이 함정이다.

## Auto Scaling

### 1. 서비스 개념

`EC2 Auto Scaling`은 인스턴스 수를 자동으로 늘리거나 줄이는 서비스다. 가용성과 탄력성을 동시에 달성하는 핵심 컴퓨트 패턴이다.

### 2. 아키텍처 구조

```text
CloudWatch Metric
      |
      v
Scaling Policy
      |
      v
Auto Scaling Group
      |
      +--> EC2 in AZ-a
      +--> EC2 in AZ-b
```

### 3. 언제 사용하는가

- 트래픽 변동이 큰 웹 애플리케이션
- 인스턴스 장애 시 자동 교체가 필요한 경우
- 일정 기반 확장과 메트릭 기반 확장을 모두 써야 할 때

### 4. 언제 사용하면 안되는가

- 단일 상태 저장 서버를 수평 확장 불가능한 구조로 운영할 때
- 스케일 아웃보다 스케일 업만 가능한 소프트웨어일 때

### 5. 비슷한 서비스와 차이

- `Auto Scaling Group`은 `EC2` 대상이다.
- `AWS Lambda`는 별도 인프라 관리 없이 자동 확장된다.
- `ECS Service Auto Scaling`은 컨테이너 태스크 수를 조정한다.

### 6. 시험에서 자주 나오는 문제 유형

- CPU 평균값 기반 확장
- 스팟과 온디맨드 혼합
- 멀티 AZ에서 최소 인스턴스 수 유지

### 7. 시험 함정 포인트

- `Auto Scaling`만으로 트래픽 분산이 되지 않는다. 앞단에 `ALB`가 필요한 경우가 많다.
- 상태 저장 세션이 인스턴스 로컬에 있으면 확장 효과가 떨어진다.

### 8. 실무 아키텍처 예시

쇼핑몰 웹 계층은 평시 최소 2대, 피크 시간에는 CPU와 요청 수 기준으로 최대 20대까지 확장한다. 인스턴스는 무상태로 설계하고 이미지 배포는 `Launch Template`으로 표준화한다.

### 시험 핵심 포인트

- 트래픽 변동과 장애 자동 복구 요구가 있으면 `Auto Scaling`을 먼저 본다.
- 무상태 애플리케이션일수록 오토스케일링 효과가 크다.
- `ALB` 없이 오토스케일링만으로 트래픽 분산이 된다고 보면 오답이 되기 쉽다.

## Launch Templates

### 1. 서비스 개념

`Launch Templates`는 `EC2` 시작 구성을 템플릿으로 저장하는 기능이다. AMI, 인스턴스 타입, 키 페어, 보안 그룹, 사용자 데이터 등을 버전 관리할 수 있다.

### 2. 아키텍처 구조

```text
Launch Template v1/v2/v3
        |
        v
Auto Scaling Group / EC2 Launch
        |
        v
Consistent Instance Configuration
```

### 3. 언제 사용하는가

- 오토스케일링 인스턴스를 일관되게 배포해야 할 때
- AMI 또는 사용자 데이터를 버전별로 관리해야 할 때
- 스팟과 온디맨드 혼합 정책을 구성할 때

### 4. 언제 사용하면 안되는가

- 수동으로 단일 테스트 인스턴스만 잠깐 띄우고 끝나는 경우
- 복잡한 배포 자동화 전체를 템플릿 하나로 해결하려는 경우

### 5. 비슷한 서비스와 차이

- 과거 `Launch Configuration`보다 `Launch Template`가 더 최신이며 기능이 많다.
- 배포 파이프라인 전체를 담당하는 서비스는 아니고, 시작 파라미터 표준화 도구다.

### 6. 시험에서 자주 나오는 문제 유형

- `Auto Scaling Group` 갱신 시 새 AMI를 템플릿 버전으로 반영하는 문제
- 스팟 인스턴스 혼합 정책을 어디에 정의하는지 묻는 문제

### 7. 시험 함정 포인트

- 오토스케일링과 함께 쓰이는 경우가 많지만 동일 서비스는 아니다.
- 버전 개념이 있어 롤백과 점진 반영에 유리하다.

### 8. 실무 아키텍처 예시

배포 파이프라인은 새 애플리케이션 AMI를 만들고 `Launch Template` 새 버전을 생성한다. 이후 `Auto Scaling Group` 인스턴스 새로 고침으로 점진 교체해 무중단 배포를 수행한다.

### 시험 핵심 포인트

- `Auto Scaling Group`의 표준화된 인스턴스 구성을 묻는 문제에서 자주 나온다.
- AMI, 사용자 데이터, 인스턴스 타입 버전 관리를 함께 생각하면 된다.
- 구형 `Launch Configuration`보다 `Launch Templates`를 우선 떠올리는 편이 안전하다.

## AWS Lambda

### 1. 서비스 개념

`AWS Lambda`는 이벤트 기반 서버리스 함수 실행 서비스다. 서버 프로비저닝 없이 코드 단위로 실행되며 사용량 기반 과금이 핵심 장점이다.

### 2. 아키텍처 구조

```text
Event Source
  |
  +--> API Gateway
  +--> S3
  +--> EventBridge
  +--> SQS
          |
          v
      AWS Lambda
          |
          v
   Downstream Service
```

### 3. 언제 사용하는가

- 간헐적 요청, 이벤트 처리, 파일 변환, 간단 API 백엔드
- 운영 부담을 줄이고 빠르게 배포해야 할 때
- 짧은 실행 시간과 자동 확장이 중요한 경우

### 4. 언제 사용하면 안되는가

- 장시간 지속 연결이나 긴 배치 작업이 필요한 경우
- 로컬 디스크 상태에 강하게 의존하는 워크로드

### 5. 비슷한 서비스와 차이

- `Amazon EC2`보다 운영 부담이 적다.
- `AWS Fargate`는 컨테이너 단위, `Lambda`는 함수 단위다.
- `Elastic Beanstalk`는 애플리케이션 플랫폼 관리, `Lambda`는 서버리스 실행이다.

### 6. 시험에서 자주 나오는 문제 유형

- `S3` 업로드 트리거 이미지 리사이징
- API 백엔드 서버리스 구현
- 큐 기반 비동기 처리

### 7. 시험 함정 포인트

- 콜드 스타트, 실행 시간 제한, 메모리/CPU 연동 특성을 이해해야 한다.
- VPC 내부 리소스 접근 시 네트워크 설정이 추가된다.

### 8. 실무 아키텍처 예시

사용자가 업로드한 이미지는 `S3` 이벤트로 `Lambda`를 호출해 썸네일을 생성하고, 메타데이터는 `DynamoDB`에 저장한다. 트래픽이 들쑥날쑥한 미디어 서비스에서 비용 효율이 높다.

### 시험 핵심 포인트

- 간헐적 요청, 이벤트 기반 처리, 서버 관리 최소화면 `AWS Lambda`가 유리하다.
- 실행 시간 제한과 콜드 스타트는 시험 함정으로 반복 출제된다.
- 장기 실행이나 복잡한 컨테이너 앱은 `AWS Fargate`나 `Amazon EC2` 쪽이 맞을 수 있다.

## Elastic Beanstalk

### 1. 서비스 개념

`AWS Elastic Beanstalk`는 애플리케이션 코드를 올리면 뒤의 `EC2`, `Auto Scaling`, `ALB` 등을 자동 구성해 주는 플랫폼 서비스다.

### 2. 아키텍처 구조

```text
Application Code
      |
      v
Elastic Beanstalk Environment
      |
      +--> ALB
      +--> Auto Scaling Group
      +--> EC2
      +--> Monitoring / Logs
```

### 3. 언제 사용하는가

- `EC2` 기반 앱을 빠르게 배포하고 싶지만 세부 인프라는 AWS가 자동화해 주길 원할 때
- 개발팀이 플랫폼 운영 부담을 줄이고 싶을 때

### 4. 언제 사용하면 안되는가

- 세밀한 인프라 제어가 매우 중요할 때
- 컨테이너 오케스트레이션이나 완전 서버리스가 더 적합할 때

### 5. 비슷한 서비스와 차이

- `Elastic Beanstalk`는 PaaS에 가깝고, `EC2` 직접 운영보다 단순하다.
- `Lambda`보다 제어권은 높지만 서버 관리가 완전히 사라지진 않는다.

### 6. 시험에서 자주 나오는 문제 유형

- 운영 복잡도를 줄이며 웹 앱을 빠르게 배포하는 선택 문제
- 기존 애플리케이션을 큰 수정 없이 플랫폼화하는 문제

### 7. 시험 함정 포인트

- 완전 서버리스가 아니다.
- 뒤에서 생성되는 리소스는 여전히 사용자의 계정 안에 존재한다.

### 8. 실무 아키텍처 예시

사내용 Java 웹 애플리케이션을 `Elastic Beanstalk`에 올려 표준화된 배포, 로그, 헬스체크, 오토스케일링 구성을 빠르게 확보한다.

### 시험 핵심 포인트

- `EC2` 기반 앱을 빠르게 배포하되 인프라 자동화를 원하면 `Elastic Beanstalk`를 본다.
- 완전 서버리스가 아니라는 점을 기억해야 한다.
- 세부 인프라 제어보다 배포 단순화가 우선일 때 적합하다.

## Amazon ECS

### 1. 서비스 개념

`Amazon ECS`는 AWS의 관리형 컨테이너 오케스트레이션 서비스다. 클러스터, 서비스, 태스크 정의를 중심으로 컨테이너를 운영한다.

### 2. 아키텍처 구조

```text
Container Image
    |
    v
ECS Task Definition
    |
    v
ECS Service
    |
    +--> ECS Cluster on EC2
    +--> ECS Cluster on Fargate
```

### 3. 언제 사용하는가

- 컨테이너 기반 마이크로서비스를 운영할 때
- 배포, 서비스 디스커버리, 오토스케일링이 필요할 때
- AWS 서비스와의 통합이 중요할 때

### 4. 언제 사용하면 안되는가

- 단일 함수 수준 이벤트 처리면 `Lambda`가 더 단순할 때
- 쿠버네티스 생태계 표준이 절대적으로 필요한 조직에서 `EKS`가 더 적합할 때

### 5. 비슷한 서비스와 차이

- `ECS`는 AWS 네이티브에 가깝고 운영이 단순하다.
- `EKS`는 Kubernetes 호환성이 강점이다.
- `Fargate`는 `ECS`의 실행 방식 중 하나다.

### 6. 시험에서 자주 나오는 문제 유형

- 마이크로서비스를 컨테이너로 분리해 운영하는 문제
- `EC2 launch type`과 `Fargate launch type` 선택 문제

### 7. 시험 함정 포인트

- `ECS`와 `Fargate`는 대체 관계가 아니라 조합 관계일 수 있다.
- 태스크 실행 역할과 태스크 역할을 혼동하면 권한 문제가 생긴다.

### 8. 실무 아키텍처 예시

주문, 결제, 재고 서비스를 각각 `ECS Service`로 배포하고, 이미지 저장소는 `ECR`, 외부 진입은 `ALB`, 비동기 이벤트는 `SQS`를 사용한다.

### 시험 핵심 포인트

- 컨테이너 오케스트레이션이 필요하면 `Amazon ECS`를 먼저 검토한다.
- `Amazon ECS`는 제어 계층이고 `AWS Fargate`는 실행 방식이라는 관계를 기억해야 한다.
- 서비스별 태스크 역할과 실행 역할을 구분하는 문제가 나올 수 있다.

## AWS Fargate

### 1. 서비스 개념

`AWS Fargate`는 서버를 직접 관리하지 않고 컨테이너를 실행하는 서버리스 컨테이너 런타임이다. `ECS` 또는 `EKS`와 함께 사용한다.

### 2. 아키텍처 구조

```text
ECS Service
   |
   v
Fargate Tasks
   |
   +--> per-task isolation
   +--> managed compute capacity
```

### 3. 언제 사용하는가

- 컨테이너는 쓰고 싶지만 EC2 클러스터 관리 부담을 줄이고 싶을 때
- 예측 불가능한 트래픽에 태스크 단위 확장이 필요할 때
- 보안 격리와 운영 단순화가 중요할 때

### 4. 언제 사용하면 안되는가

- 매우 세밀한 OS 제어가 필요한 경우
- 상시 고정 부하에서 `EC2` 기반이 더 저렴한 경우

### 5. 비슷한 서비스와 차이

- `ECS on EC2`는 비용 최적화와 커스터마이징에 유리할 수 있다.
- `Fargate`는 운영 단순성이 강점이다.
- `Lambda`보다 긴 실행, 다중 컨테이너 패턴에 유리하다.

### 6. 시험에서 자주 나오는 문제 유형

- 서버 관리 부담 최소화가 요구되는 컨테이너 문제
- 예측 어려운 트래픽에서 컨테이너 앱을 빠르게 배포하는 문제

### 7. 시험 함정 포인트

- 서버리스라고 해서 모든 비용이 더 싸다는 뜻은 아니다.
- 실행 시간, 리소스 단위, 네트워크 비용을 함께 봐야 한다.

### 8. 실무 아키텍처 예시

이벤트 기반 API 백엔드를 `ECS on Fargate`로 운영해 배포 표준화와 오토스케일링을 확보한다. 운영팀은 클러스터 노드 패치 대신 애플리케이션 태스크 버전 관리에 집중할 수 있다.

### 시험 핵심 포인트

- 컨테이너는 필요하지만 서버는 관리하고 싶지 않을 때 `AWS Fargate`가 적합하다.
- 서버리스라고 항상 가장 저렴한 것은 아니므로 상시 부하는 `EC2`와 비교해야 한다.
- `Amazon ECS`와 함께 등장할 때는 실행 런타임 관점으로 읽어야 한다.

## 장 마무리

- 운영 제어권이 중요하면 `Amazon EC2`, 운영 단순성이 중요하면 `AWS Lambda` 또는 `AWS Fargate` 쪽으로 기울면 된다.
- 오토스케일링은 무상태 설계와 함께 써야 효과가 크다.
- 컨테이너 문제에서는 `Amazon ECS`와 `AWS Fargate`의 관계를 정확히 구분해야 한다.
