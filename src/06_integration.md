## Amazon SQS

### 1. 서비스 개념

`Amazon SQS`는 완전관리형 메시지 큐 서비스다. 생산자와 소비자를 느슨하게 결합하고, 트래픽 버스트를 흡수하는 데 적합하다.

### 2. 아키텍처 구조

```text
Producer
  |
  v
SQS Queue
  |
  v
Consumer Fleet / Lambda
```

### 3. 언제 사용하는가

- 비동기 처리로 시스템 결합도를 낮추고 싶을 때
- 작업량이 급증해도 큐에 적재해 소비 속도를 조절해야 할 때
- 장애 격리와 재시도 구조가 필요할 때

### 4. 언제 사용하면 안되는가

- 메시지 순서를 엄격히 보장해야 하는데 표준 큐를 사용하는 경우
- 실시간 다중 구독 팬아웃이 필요한 경우

### 5. 비슷한 서비스와 차이

- `SNS`는 푸시형 팬아웃, `SQS`는 폴링형 큐다.
- `EventBridge`는 이벤트 라우팅 버스다.

### 6. 시험에서 자주 나오는 문제 유형

- 웹 요청 처리 후 백엔드 비동기 작업
- 버스트 흡수
- DLQ와 재처리

### 7. 시험 함정 포인트

- 중복 처리 가능성을 염두에 둔 idempotent 설계가 필요하다.
- 메시지는 소비 후 삭제해야 한다.

### 8. 실무 아키텍처 예시

주문 생성 후 이메일 발송, 재고 갱신, 송장 생성 작업을 큐에 넣고 각각의 소비자가 독립 처리한다. 프런트엔드 응답 속도와 백엔드 안정성을 함께 확보할 수 있다.

### 시험 핵심 포인트

- 애플리케이션 컴포넌트 분리와 비동기 버퍼링은 `Amazon SQS`의 대표 시나리오다.
- 버스트 흡수, 재시도, DLQ가 핵심 포인트다.
- 팬아웃은 `Amazon SNS`, 규칙 기반 라우팅은 `Amazon EventBridge`와 비교해야 한다.

## Amazon SQS FIFO

### 1. 서비스 개념

`Amazon SQS FIFO`는 순서 보장과 중복 제거를 제공하는 큐다. 표준 큐보다 처리량 특성이 다르다.

### 2. 아키텍처 구조

```text
Producer
  |
  v
FIFO Queue
  |
  +--> Message Group ID
  +--> Deduplication ID
  |
  v
Ordered Consumer Processing
```

### 3. 언제 사용하는가

- 주문 이벤트처럼 순서가 깨지면 안 되는 경우
- 중복 처리 방지가 중요한 경우

### 4. 언제 사용하면 안되는가

- 초대규모 병렬 처리량이 최우선인데 순서가 중요하지 않은 경우

### 5. 비슷한 서비스와 차이

- `SQS Standard`는 거의 무제한 처리량과 at-least-once
- `SQS FIFO`는 순서 보장과 정확히 한 번처럼 보이는 처리 보조 기능 제공

### 6. 시험에서 자주 나오는 문제 유형

- 순서 보장 메시징
- 중복 없는 거래 이벤트 처리

### 7. 시험 함정 포인트

- 순서 요구사항이 있으면 FIFO를 고려해야 하지만 처리량 제약을 함께 본다.
- 메시지 그룹 ID 단위 순서를 이해해야 한다.

### 8. 실무 아키텍처 예시

결제 상태 변경 이벤트는 동일 주문 ID를 메시지 그룹으로 사용해 순차 처리한다. 결제 승인 전에 취소가 적용되는 오류를 방지할 수 있다.

### 시험 핵심 포인트

- 메시지 순서 보장과 중복 제거가 필요하면 `Amazon SQS FIFO`를 선택한다.
- 표준 큐보다 처리량 특성이 다르므로 요구사항 문장을 잘 읽어야 한다.
- 메시지 그룹 ID 개념이 자주 함정으로 나온다.

## Amazon SNS

### 1. 서비스 개념

`Amazon SNS`는 퍼블리시/서브스크라이브 기반 메시징 서비스다. 하나의 이벤트를 여러 구독자에게 팬아웃하는 데 적합하다.

### 2. 아키텍처 구조

```text
Publisher
  |
  v
SNS Topic
  |
  +--> SQS Queue A
  +--> SQS Queue B
  +--> Lambda
  +--> Email / SMS
```

### 3. 언제 사용하는가

- 동일 이벤트를 여러 시스템에 동시에 전달해야 할 때
- 알림, 팬아웃, 이벤트 방송 구조가 필요할 때

### 4. 언제 사용하면 안되는가

- 소비자가 자기 속도로 처리해야 하는 큐 기반 버퍼링이 핵심일 때
- 복잡한 규칙 기반 이벤트 라우팅이 필요할 때

### 5. 비슷한 서비스와 차이

- `SNS`는 팬아웃
- `SQS`는 버퍼링
- `EventBridge`는 규칙 기반 라우팅

### 6. 시험에서 자주 나오는 문제 유형

- 주문 이벤트를 여러 후속 시스템에 동시에 전달
- 알림을 여러 채널로 전송

### 7. 시험 함정 포인트

- 신뢰성 있는 팬아웃은 `SNS -> SQS` 조합이 자주 정답이다.
- 단일 소비자가 아니라 다중 소비자 문제인지 먼저 확인해야 한다.

### 8. 실무 아키텍처 예시

회원 가입 이벤트를 `SNS Topic`에 발행하고, 환영 메일 서비스, CRM 적재 서비스, 분석 파이프라인이 각자 구독해 처리한다.

### 시험 핵심 포인트

- 하나의 이벤트를 여러 구독자에게 동시에 전달하는 문제는 `Amazon SNS`가 강하다.
- 신뢰성 있는 팬아웃은 `SNS -> SQS` 조합으로 자주 출제된다.
- 버퍼링이 아니라 푸시형 알림/배포라는 점을 기억해야 한다.

## Amazon EventBridge

### 1. 서비스 개념

`Amazon EventBridge`는 이벤트 버스 서비스다. SaaS, AWS 서비스, 애플리케이션 커스텀 이벤트를 규칙 기반으로 여러 대상에 전달한다.

### 2. 아키텍처 구조

```text
Event Sources
   |
   v
EventBridge Bus
   |
   +--> Rule 1 -> Lambda
   +--> Rule 2 -> SQS
   +--> Rule 3 -> Step Functions
```

### 3. 언제 사용하는가

- 이벤트 기반 아키텍처를 규칙 중심으로 구성할 때
- 여러 서비스에서 발생한 이벤트를 중앙 버스에서 분기할 때
- AWS 서비스 상태 변화를 자동화 트리거로 사용할 때

### 4. 언제 사용하면 안되는가

- 순서 보장 큐가 필요한 경우
- 대량 스트림 데이터 처리 자체가 핵심인 경우

### 5. 비슷한 서비스와 차이

- `SNS`는 단순 팬아웃
- `EventBridge`는 조건 기반 라우팅, 스키마, SaaS 통합이 강점이다

### 6. 시험에서 자주 나오는 문제 유형

- EC2 상태 변화 이벤트로 자동화 실행
- 여러 마이크로서비스 이벤트 분기
- 크론 스케줄 기반 이벤트

### 7. 시험 함정 포인트

- 큐가 아니라 이벤트 버스다.
- 소비자 재처리 버퍼가 필요하면 `SQS`를 대상에 붙여야 한다.

### 8. 실무 아키텍처 예시

결제 완료 이벤트는 `EventBridge`에서 규칙별로 청구서 생성, 포인트 적립, 분석 적재 워크플로로 분기한다. 서비스 간 직접 호출보다 결합도가 낮다.

### 시험 핵심 포인트

- 규칙 기반 이벤트 라우팅과 AWS 서비스 이벤트 처리에는 `Amazon EventBridge`가 적합하다.
- 큐가 아니라 이벤트 버스이므로 재처리 버퍼가 필요하면 `Amazon SQS`를 붙인다.
- SaaS 통합과 스케줄 이벤트도 대표 출제 포인트다.

## AWS Step Functions

### 1. 서비스 개념

`AWS Step Functions`는 여러 작업 단계를 상태 머신으로 조정하는 워크플로 서비스다. 순차, 병렬, 재시도, 예외 처리에 강하다.

### 2. 아키텍처 구조

```text
Start
  |
  v
Task 1 -> Choice -> Parallel -> Task 2 -> Success/Fail
```

### 3. 언제 사용하는가

- 여러 `Lambda`나 서비스 호출을 순서대로 조정할 때
- 장시간 실행 비즈니스 프로세스를 시각적으로 관리할 때
- 재시도와 오류 분기가 중요한 경우

### 4. 언제 사용하면 안되는가

- 단순 단일 함수 호출이면 오버엔지니어링일 때
- 초저지연 실시간 메시지 브로커로 쓰려는 경우

### 5. 비슷한 서비스와 차이

- `Lambda`는 실행 단위, `Step Functions`는 오케스트레이션
- `EventBridge`는 이벤트 분기, `Step Functions`는 상태 있는 절차 제어

### 6. 시험에서 자주 나오는 문제 유형

- 주문 승인, 결제, 배송 생성 같은 다단계 작업
- 재시도/백오프/실패 처리 자동화

### 7. 시험 함정 포인트

- 비동기 이벤트 전달과 상태 기반 오케스트레이션을 구분해야 한다.
- 사람 승인 단계처럼 긴 프로세스도 적합할 수 있다.

### 8. 실무 아키텍처 예시

대출 심사 시스템은 서류 검증, 신용 평가, 수동 승인, 계약 생성 단계를 `Step Functions`로 표현한다. 운영자는 실패 지점을 쉽게 추적할 수 있다.

### 시험 핵심 포인트

- 여러 단계를 가진 상태 기반 워크플로는 `AWS Step Functions`를 먼저 본다.
- 재시도, 오류 분기, 장시간 프로세스 제어가 장점이다.
- 단순 이벤트 전달과는 목적이 다르다는 점을 `Amazon EventBridge`와 구분해야 한다.

## Amazon Kinesis

### 1. 서비스 개념

`Amazon Kinesis`는 실시간 스트리밍 데이터 수집과 처리를 위한 서비스군이다. 시험에서는 보통 `Kinesis Data Streams`, `Firehose` 관점으로 등장한다.

### 2. 아키텍처 구조

```text
Producers
  |
  v
Kinesis Data Streams
  |
  +--> Consumer App
  +--> Lambda
  +--> Analytics
```

### 3. 언제 사용하는가

- 클릭스트림, IoT, 로그처럼 연속 이벤트를 실시간 처리할 때
- 대량 스트림을 여러 소비자가 독립 처리할 때

### 4. 언제 사용하면 안되는가

- 단순 비동기 작업 큐면 `SQS`가 더 단순한 경우
- 순차 워크플로 제어가 필요하면 `Step Functions`가 맞는 경우

### 5. 비슷한 서비스와 차이

- `SQS`는 큐, `Kinesis`는 스트림
- `Firehose`는 목적지 적재 자동화에 강하고, `Data Streams`는 커스텀 실시간 처리에 강하다

### 6. 시험에서 자주 나오는 문제 유형

- 실시간 로그 분석
- 스트리밍 데이터 다중 소비
- 샤드 확장과 처리량

### 7. 시험 함정 포인트

- 스트림 보존 기간과 재처리 가능성을 이해해야 한다.
- 큐와 달리 여러 소비자가 같은 데이터를 각각 읽을 수 있다.

### 8. 실무 아키텍처 예시

모바일 앱 이벤트를 `Kinesis Data Streams`로 수집하고, 한 소비자는 실시간 대시보드, 다른 소비자는 장기 저장 파이프라인으로 보낸다.

### 시험 핵심 포인트

- 실시간 스트리밍 데이터 수집과 다중 소비자 처리는 `Amazon Kinesis`가 적합하다.
- 큐와 달리 여러 소비자가 같은 스트림을 독립적으로 읽을 수 있다.
- 대량 연속 이벤트라는 단서가 보이면 `Amazon SQS`보다 우선순위가 올라간다.

## 장 마무리

- 비동기 버퍼링은 `SQS`, 팬아웃은 `SNS`, 규칙 라우팅은 `EventBridge`, 절차 제어는 `Step Functions`, 실시간 스트림은 `Kinesis`로 정리하면 된다.
- 순서 보장 요구사항이 보이면 `SQS FIFO`를 먼저 검토한다.
