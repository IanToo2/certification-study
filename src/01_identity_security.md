## 빠른 관점 정리

- 인증은 `누구인가`, 인가는 `무엇을 할 수 있는가`다.
- 사람 사용자 권한은 `IAM Identity Center` 또는 역할 위임 관점으로 생각하고, 애플리케이션 권한은 `IAM Role`로 보는 습관이 중요하다.
- 데이터 보호는 `KMS`, 비밀값 보호는 `Secrets Manager`, 네트워크 레벨 차단은 `Security Group`과 `Network ACL`로 나눠 본다.

## IAM

### 1. 서비스 개념

`IAM`은 AWS 계정 안에서 사용자, 그룹, 역할, 정책을 이용해 권한을 제어하는 핵심 서비스다. 시험에서는 거의 모든 서비스 시나리오의 전제 조건으로 등장하며, 최소 권한 원칙을 구현하는 기본 도구로 취급된다.

### 2. 아키텍처 구조

```text
Developer / Application
        |
        v
      IAM
  (User / Group / Role)
        |
        v
   Policy Evaluation
        |
        v
 AWS Resource Access
```

핵심은 인증 주체와 정책이 분리된다는 점이다. 사용자나 역할 자체가 권한을 가지는 것이 아니라, 연결된 정책 평가 결과로 허용 여부가 결정된다.

### 3. 언제 사용하는가

- 계정 내 사용자 또는 워크로드 권한을 중앙에서 통제해야 할 때
- 서비스 간 호출 권한을 안전하게 위임해야 할 때
- 리소스별 세밀한 접근 제어가 필요할 때

### 4. 언제 사용하면 안되는가

- 애플리케이션 내부 사용자 인증을 직접 구현하려는 경우
- 장기 액세스 키를 애플리케이션 서버에 고정 저장하는 방식으로 운영하려는 경우

### 5. 비슷한 서비스와 차이

- `IAM`과 `Amazon Cognito`: `IAM`은 AWS 리소스 권한 관리, `Amazon Cognito`는 애플리케이션 최종 사용자 인증에 가깝다.
- `IAM`과 `AWS Organizations SCP`: `IAM`은 계정 내부 허용/거부, `SCP`는 계정 또는 OU 단위의 상한선 제어다.

### 6. 시험에서 자주 나오는 문제 유형

- 특정 서비스만 접근 가능하도록 최소 권한 정책을 설계하는 문제
- 서비스가 다른 서비스에 접근해야 할 때 사용자 키 대신 역할을 선택하는 문제
- 명시적 거부가 허용보다 우선한다는 정책 평가 문제

### 7. 시험 함정 포인트

- 루트 계정 사용은 예외적 작업에만 제한해야 한다.
- `Allow`가 있어도 `Explicit Deny`가 있으면 거부된다.
- 권한 부족 문제를 액세스 키 발급으로 해결하려 하면 대부분 오답이다.

### 8. 실무 아키텍처 예시

운영자는 `IAM Identity Center`로 콘솔에 로그인하고, `EC2` 인스턴스는 `IAM Role`을 통해 `S3`와 `CloudWatch`에 접근한다. 이렇게 하면 사람과 시스템의 권한 경계를 분리하면서 키 회전 부담을 줄일 수 있다.

### 시험 핵심 포인트

- AWS 리소스 권한 문제의 기본 정답 축은 `IAM`이다.
- 정답 판단 시 최소 권한과 명시적 거부 우선 원칙을 함께 본다.
- 애플리케이션 최종 사용자 로그인 문제는 `Amazon Cognito`와 구분해야 한다.

## IAM Roles

### 1. 서비스 개념

`IAM Role`은 장기 자격 증명 없이 임시 권한을 위임하는 메커니즘이다. `EC2`, `Lambda`, `ECS Task`, 교차 계정 접근 등 거의 모든 현대적 AWS 권한 설계의 기본 단위다.

### 2. 아키텍처 구조

```text
Application
   |
   v
Assume Role / Attached Role
   |
   v
STS Temporary Credentials
   |
   v
Target AWS Service
```

`STS`가 발급하는 임시 자격 증명이 핵심이며, 만료 시간이 있어 장기 키보다 안전하다.

### 3. 언제 사용하는가

- `EC2`, `Lambda`, `ECS`, `Step Functions`가 다른 서비스에 접근할 때
- 교차 계정에서 감사 또는 운영 역할을 위임할 때
- 사람 사용자가 특정 작업 시점에만 승격 권한을 얻어야 할 때

### 4. 언제 사용하면 안되는가

- 동일 계정의 사람 사용자에게 상시 고정 권한을 주려는 경우
- 외부 사용자 인증 자체를 `IAM Role`로 해결하려는 경우

### 5. 비슷한 서비스와 차이

- `IAM User`와 비교하면 `IAM Role`은 장기 비밀키가 없고 세션 기반이다.
- `Instance Profile`은 `EC2`에 역할을 연결하는 컨테이너 개념이지 별도 권한 모델이 아니다.

### 6. 시험에서 자주 나오는 문제 유형

- `EC2`가 `S3`에 접근해야 할 때 액세스 키를 파일에 저장하지 않고 역할을 붙이는 문제
- 교차 계정 버킷 접근을 위해 역할 신뢰 정책을 구성하는 문제

### 7. 시험 함정 포인트

- 역할에는 권한 정책과 신뢰 정책이 모두 필요하다.
- 교차 계정 문제에서 권한 정책만 보고 신뢰 정책을 빼먹으면 오답이다.
- `EC2`에 역할을 부여할 때 실제로는 인스턴스 프로파일이 연결된다.

### 8. 실무 아키텍처 예시

보안 계정의 감사 시스템이 각 워크로드 계정의 읽기 전용 역할을 `AssumeRole`해 로그와 설정 상태를 수집한다. 이 구조는 중앙 감사에 자주 쓰이며, SAA에서도 교차 계정 베스트 프랙티스로 자주 등장한다.

### 시험 핵심 포인트

- 서비스 간 권한 위임은 장기 액세스 키보다 `IAM Role`이 우선이다.
- `EC2`, `AWS Lambda`, `Amazon ECS` 권한 부여 문제에서 가장 자주 나온다.
- 교차 계정 문제는 권한 정책과 신뢰 정책을 함께 확인해야 한다.

## IAM Policies

### 1. 서비스 개념

`IAM Policy`는 JSON 문서 형태의 권한 정의다. 어떤 액션을 어떤 리소스에 대해 허용 또는 거부하는지 기술한다.

### 2. 아키텍처 구조

```text
Principal
  |
  +--> Identity-based Policy
  |
  +--> Resource-based Policy
  |
  +--> Permissions Boundary / SCP
            |
            v
     Final Authorization Decision
```

시험에서는 여러 정책 계층이 동시에 평가된다는 점이 중요하다.

### 3. 언제 사용하는가

- 서비스별 최소 권한 정책을 설계할 때
- 특정 버킷, 큐, 키에 대한 제한된 접근을 정의할 때
- 태그 기반 권한 제어를 할 때

### 4. 언제 사용하면 안되는가

- 사람에게 설명 가능한 단순 운영 절차 대신 과도하게 복잡한 조건문을 넣으려는 경우
- 계정 전체 상한선을 `IAM Policy`만으로 강제하려는 경우

### 5. 비슷한 서비스와 차이

- `Identity-based Policy`는 사용자/역할에 붙는다.
- `Resource-based Policy`는 `S3`, `SQS`, `KMS` 같은 리소스에 직접 붙는다.
- `SCP`는 허용 상한선이고, 계정 멤버가 벗어날 수 없다.

### 6. 시험에서 자주 나오는 문제 유형

- 특정 IP나 태그 조건으로 접근을 제한하는 문제
- `S3 Bucket Policy`와 `IAM Policy`를 조합해 교차 계정 접근을 구성하는 문제

### 7. 시험 함정 포인트

- 정책 시뮬레이터 없이도 논리 흐름을 읽을 수 있어야 한다.
- 리소스 기반 정책이 가능한 서비스와 불가능한 서비스를 구분해야 한다.
- 와일드카드 사용은 편하지만 시험에서는 최소 권한 위배로 오답 유도에 쓰인다.

### 8. 실무 아키텍처 예시

운영팀 역할에는 읽기 전용 공통 정책을 붙이고, 프로덕션 수정 권한은 별도 역할에 한정한다. `S3 Bucket Policy`는 특정 역할만 수용하도록 제한해 계정 내 과도한 권한 확산을 막는다.

### 시험 핵심 포인트

- 정책 평가 문제는 `Allow`보다 `Explicit Deny`가 우선한다.
- 리소스 기반 정책을 지원하는 서비스인지 먼저 구분해야 한다.
- 시험에서는 와일드카드보다 최소 권한 정책이 정답 후보가 된다.

## KMS

### 1. 서비스 개념

`AWS KMS`는 데이터 암호화 키를 중앙에서 생성, 저장, 사용 통제하는 관리형 키 서비스다. SAA에서는 저장 데이터 암호화, 서비스 통합, 키 정책 이해가 핵심이다.

### 2. 아키텍처 구조

```text
Application / AWS Service
        |
        v
       KMS
   (CMK / Key Policy)
        |
        v
Envelope Encryption
        |
        v
Encrypted Data at Rest
```

실제 대용량 데이터는 데이터 키로 암호화하고, 데이터 키는 `KMS key`로 다시 보호하는 envelope encryption 구조가 기본이다.

### 3. 언제 사용하는가

- `S3`, `EBS`, `RDS`, `EFS`, `Secrets Manager` 등과 통합 암호화가 필요할 때
- 키 사용 권한과 감사 추적을 중앙 관리해야 할 때
- 고객 관리형 키를 사용해 접근 통제를 세밀하게 해야 할 때

### 4. 언제 사용하면 안되는가

- 애플리케이션 내부 단순 해시 저장을 키 관리 문제와 혼동하는 경우
- 온프레미스 HSM 전용 요구사항을 `KMS`만으로 충족하려는 경우

### 5. 비슷한 서비스와 차이

- `KMS`와 `CloudHSM`: `KMS`는 관리형, `CloudHSM`은 전용 HSM 제어권이 더 크다.
- `AWS managed key`와 `Customer managed key`: 고객 관리형 키가 권한 제어와 회전, 정책 통제가 더 유연하다.

### 6. 시험에서 자주 나오는 문제 유형

- `S3 SSE-KMS`, `EBS encryption`, `RDS encryption` 설정 문제
- 교차 계정 암호화된 객체 접근 시 키 정책과 IAM 권한을 함께 묻는 문제

### 7. 시험 함정 포인트

- `KMS` 권한이 없으면 암호화된 데이터 리소스 권한이 있어도 접근하지 못할 수 있다.
- `AWS managed key`는 교차 계정 공유 요구사항에 제약이 있을 수 있다.
- 암호화는 리전 단위라는 점을 잊으면 복제 시나리오에서 헷갈린다.

### 8. 실무 아키텍처 예시

민감 정보가 저장된 `Amazon S3` 버킷과 `Amazon RDS`는 고객 관리형 `KMS key`를 사용한다. 감사팀은 키 사용 로그를 `CloudTrail`로 추적해 누가 언제 복호화 권한을 사용했는지 검토한다.

### 시험 핵심 포인트

- 저장 데이터 암호화가 보이면 `AWS KMS` 통합 가능 여부를 먼저 본다.
- 암호화된 리소스 접근은 리소스 권한과 키 권한이 모두 필요할 수 있다.
- 교차 계정 암호화 문제는 키 정책을 빼먹기 쉽다.

## Secrets Manager

### 1. 서비스 개념

`AWS Secrets Manager`는 데이터베이스 비밀번호, API 키, 토큰 같은 비밀값을 안전하게 저장하고 자동 회전까지 지원하는 서비스다.

### 2. 아키텍처 구조

```text
Application
   |
   v
Secrets Manager
   |
   +--> KMS Encryption
   |
   +--> Rotation Lambda
   |
   v
Database / External API Credential
```

### 3. 언제 사용하는가

- 비밀번호를 코드나 환경 변수에 평문 저장하면 안 될 때
- `RDS` 자격 증명을 자동 회전해야 할 때
- 여러 애플리케이션이 동일한 비밀값을 중앙 참조해야 할 때

### 4. 언제 사용하면 안되는가

- 단순 설정값까지 모두 비밀값 저장소에 넣어 비용을 늘리는 경우
- 장기 파일 형태 설정을 버전 관리 없이 무조건 비밀값으로만 처리하려는 경우

### 5. 비슷한 서비스와 차이

- `Systems Manager Parameter Store`는 일반 설정과 저비용 저장에 적합하다.
- `Secrets Manager`는 자동 회전, 비밀값 생명주기 관리에 더 적합하다.

### 6. 시험에서 자주 나오는 문제 유형

- 애플리케이션 비밀번호를 안전하게 저장하고 자동 회전해야 하는 문제
- `EC2` 또는 `Lambda`가 비밀값을 조회할 때 역할 기반 접근을 구성하는 문제

### 7. 시험 함정 포인트

- 환경 변수에 암호문을 넣는 것과 비밀값 관리 서비스 사용은 다르다.
- 자동 회전이 필요한 요구사항이면 `Secrets Manager` 쪽이 정답일 가능성이 높다.
- 저장만 안전해도, 가져가는 주체에 `IAM` 권한이 없으면 동작하지 않는다.

### 8. 실무 아키텍처 예시

`AWS Lambda` 기반 API가 `Amazon RDS`에 접속할 때 비밀번호를 코드에 넣지 않고 `Secrets Manager`에서 조회한다. 데이터베이스 비밀번호는 회전용 `Lambda`로 주기적으로 갱신해 유출 위험을 줄인다.

### 시험 핵심 포인트

- 비밀번호, 토큰, API 키와 자동 회전 요구사항이 있으면 `Secrets Manager`가 강한 정답 후보다.
- 환경 변수 평문 저장보다 중앙 비밀 저장소가 우선이다.
- `IAM Role`과 함께 써야 안전하게 조회할 수 있다.

## Amazon Cognito

### 1. 서비스 개념

`Amazon Cognito`는 웹과 모바일 애플리케이션의 사용자 가입, 로그인, 토큰 발급, 연합 인증을 제공하는 서비스다. 시험에서는 애플리케이션 최종 사용자 인증이 필요할 때 자주 정답 후보가 된다.

### 2. 아키텍처 구조

```text
End User
   |
   v
Amazon Cognito User Pool
   |
   +--> Authentication / Federation
   |
   v
JWT Token
   |
   v
App / API Gateway / ALB
```

필요하면 `Identity Pool`을 통해 AWS 자격 증명 위임까지 연결할 수 있다.

### 3. 언제 사용하는가

- 모바일 앱이나 웹 앱의 사용자 로그인 기능이 필요할 때
- 소셜 로그인이나 SAML/OIDC 연동이 필요할 때
- 인증 후 제한된 AWS 리소스 접근을 사용자별로 제어해야 할 때

### 4. 언제 사용하면 안되는가

- AWS 계정 내부 관리자 권한 통제를 구현하려는 경우
- 단순 서비스 간 권한 위임을 최종 사용자 인증과 혼동하는 경우

### 5. 비슷한 서비스와 차이

- `IAM`은 AWS 리소스 관리자 권한, `Amazon Cognito`는 애플리케이션 사용자 인증이다.
- `Cognito User Pool`은 인증, `Identity Pool`은 AWS 자격 증명 브로커 역할이다.

### 6. 시험에서 자주 나오는 문제 유형

- 모바일 앱 사용자가 업로드한 파일을 사용자별로 분리 저장하는 시나리오
- 소셜 로그인과 토큰 기반 인증이 필요한 서버리스 앱 시나리오

### 7. 시험 함정 포인트

- 로그인 자체가 필요한 문제인데 `IAM User`를 답으로 고르면 오답이다.
- 인증과 권한 부여를 함께 보지 않으면 `User Pool`과 `Identity Pool` 선택이 흔들린다.

### 8. 실무 아키텍처 예시

모바일 앱 사용자는 `Amazon Cognito User Pool`로 로그인하고, 인증된 사용자는 `Identity Pool`을 통해 제한된 `S3` 업로드 권한을 받는다. 이 패턴은 서버리스 이미지 업로드 앱에서 매우 흔하다.

### 시험 핵심 포인트

- 최종 사용자 회원가입과 로그인 문제는 `Amazon Cognito`를 먼저 검토한다.
- `IAM`은 AWS 관리자/서비스 권한이고 `Amazon Cognito`는 애플리케이션 사용자 인증이다.
- 사용자 인증 뒤 제한된 AWS 접근이 필요하면 Identity Pool 연결 여부를 본다.

## Security Groups

### 1. 서비스 개념

`Security Group`은 인스턴스나 ENI 수준에 적용되는 상태 저장(stateful) 가상 방화벽이다. 허용 규칙만 정의하며 응답 트래픽은 자동 허용된다.

### 2. 아키텍처 구조

```text
Client
  |
  v
ALB Security Group
  |
  v
EC2 Security Group
  |
  v
RDS Security Group
```

보통 계층별로 보안 그룹을 분리하고, 소스에 CIDR 대신 다른 보안 그룹을 참조해 애플리케이션 흐름을 정의한다.

### 3. 언제 사용하는가

- `EC2`, `ALB`, `RDS`, `EFS` 등 리소스별 인바운드/아웃바운드 제어가 필요할 때
- 애플리케이션 계층 간 통신을 논리적으로 제한할 때

### 4. 언제 사용하면 안되는가

- 서브넷 단위 거부 규칙이 필요한 경우
- 네트워크 경계 전체에 대한 순서 기반 허용/거부가 필요한 경우

### 5. 비슷한 서비스와 차이

- `Security Group`은 상태 저장, `Network ACL`은 상태 비저장이다.
- `Security Group`은 인스턴스 수준, `Network ACL`은 서브넷 수준이다.

### 6. 시험에서 자주 나오는 문제 유형

- 웹 서버는 80/443만 열고 데이터베이스는 웹 서버 보안 그룹에서만 3306 허용하는 문제
- `ALB`와 백엔드 인스턴스의 포트 흐름을 설계하는 문제

### 7. 시험 함정 포인트

- 거부 규칙은 없다.
- 응답 패킷은 상태 저장 특성 때문에 자동 허용된다.
- 잘못된 0.0.0.0/0 허용은 시험에서 자주 오답 유도에 쓰인다.

### 8. 실무 아키텍처 예시

퍼블릭 서브넷의 `ALB`는 443만 인터넷에 노출한다. 프라이빗 서브넷의 `EC2`는 `ALB Security Group`에서만 8080 접근을 허용하고, `RDS`는 애플리케이션 보안 그룹에서만 3306을 받는다.

### 시험 핵심 포인트

- 인스턴스 단위 접근 제어는 `Security Group`이 기본이다.
- 상태 저장이므로 응답 트래픽은 별도 허용 없이 통과한다.
- 거부 규칙이 필요하면 `Network ACL`과 비교해야 한다.

## Network ACL

### 1. 서비스 개념

`Network ACL`은 서브넷 단위로 적용되는 상태 비저장(stateless) 네트워크 필터다. 허용과 거부 규칙을 번호 순서대로 평가한다.

### 2. 아키텍처 구조

```text
Internet
   |
   v
Subnet Boundary
   |
   v
Network ACL
   |
   v
Instances in Subnet
```

### 3. 언제 사용하는가

- 서브넷 단위로 특정 IP 대역을 차단해야 할 때
- 보안 그룹 외에 추가적인 네트워크 레이어 방어가 필요할 때

### 4. 언제 사용하면 안되는가

- 인스턴스별 세밀한 허용 제어만 필요할 때
- 상태 저장 연결을 기대하며 응답 트래픽 규칙을 생략하려는 경우

### 5. 비슷한 서비스와 차이

- `Security Group`은 인스턴스 기반, 상태 저장
- `Network ACL`은 서브넷 기반, 상태 비저장, 명시적 거부 가능

### 6. 시험에서 자주 나오는 문제 유형

- 특정 악성 IP 범위를 빠르게 차단하는 문제
- 퍼블릭/프라이빗 서브넷 경계에서 추가 제어를 요구하는 문제

### 7. 시험 함정 포인트

- 응답 트래픽을 위한 ephemeral port 허용을 잊으면 통신이 실패한다.
- 낮은 번호 규칙이 먼저 평가된다.
- 보안 그룹 대체제가 아니라 보완재다.

### 8. 실무 아키텍처 예시

인터넷 노출 퍼블릭 서브넷에는 `Network ACL`로 비정상 국가 IP 대역을 차단하고, 실제 애플리케이션 접근 제어는 `Security Group`으로 세밀하게 수행한다. 시험에서는 이 이중 방어 구조를 이해하면 선택지가 쉽게 정리된다.

### 시험 핵심 포인트

- 서브넷 경계 차단 문제와 명시적 거부 규칙은 `Network ACL`이 맞다.
- 상태 비저장이므로 반환 트래픽용 ephemeral port를 함께 고려해야 한다.
- `Security Group` 대체가 아니라 보완 레이어로 출제된다.

## 장 마무리

- 사람과 시스템 권한은 분리하고, 시스템에는 항상 `IAM Role`을 우선 적용한다.
- 암호화 문제는 `KMS`, 비밀값 문제는 `Secrets Manager`, 최종 사용자 로그인 문제는 `Amazon Cognito`를 우선 떠올린다.
- 네트워크 제어는 `Security Group`과 `Network ACL`의 상태 저장 여부 차이를 확실히 구분해야 한다.
